# 风上忍小说AI Agent 调研报告

> 调研日期：2026-03-30  
> 调研人：虾虾子（AI Agent架构调研）  
> 目标：开发一个能模仿风上忍风格写中文轻小说的AI Agent

---

## 一、调研概述

本报告旨在系统调研开发"风上忍小说AI Agent"所需的技术架构、核心组件、实施路径和风险评估。目标是基于已有的5本小说全文（1100万字）、风格分析报告、RAG系统等资源，构建一个能持续输出风格一致的长篇轻小说的多Agent写作系统。

调研覆盖以下方向：
- 主流AI写作Agent方案和开源项目
- 多Agent协作写作架构
- RAG + LLM 在创意写作中的最佳实践
- 风格控制、记忆管理、质量控制等关键技术
- 基于OpenClaw环境和mimo-v2-pro模型的落地可行性

## 二、现有方案分析

### 2.1 顶级开源项目（按成熟度排序）

#### ① InkOS（⭐2.9k，最成熟）
- **架构**：10-Agent管线：Radar → Planner → Composer → Architect → Writer → Observer → Reflector → Normalizer → Auditor → Reviser
- **核心创新**：
  - **7个真相文件**（JSON + Zod校验）：世界状态、资源账本、伏笔系统、章节摘要、支线进度、情感弧线、角色矩阵
  - **JSON delta写入**：不输出全量markdown，而是输出结构化delta，避免上下文膨胀
  - **33维度连续性审计**：角色记忆、物资连续性、伏笔回收、AI味检测等
  - **伏笔治理**：hookAgenda排班 + analyzeHookHealth审计 + evaluateHookAdmission拦截重复
  - **字数治理**：LengthSpec + Normalizer单pass纠偏
  - **SQLite时序记忆**：Node 22+环境按相关性检索历史
  - **风格指纹注入**：从参考文本提取句长分布、词频特征、节奏模式
  - **已发布为OpenClaw Skill**：可直接通过clawhub安装
- **技术栈**：TypeScript/Node.js，OpenAI兼容接口，支持多模型路由
- **对我们的价值**：**架构参考价值极高**，其Agent管线设计、真相文件系统、审计机制可直接借鉴

#### ② NovelClaw（⭐147，哈工大深圳）
- **架构**：动态记忆优先（Dynamic-Memory-First）的协作框架
- **核心创新**：
  - 可编辑记忆库（memory banks）——作者可以手动修正AI的记忆
  - 会话式写作工作区（非一次性生成）
  - 故事板、角色面板、世界观面板持续可见
  - 运行日志可审查
- **技术栈**：Python + FastAPI
- **对我们的价值**：其"可编辑记忆"概念值得借鉴，但架构不如InkOS成熟

#### ③ AI-Novel-Writing-Assistant（⭐33，中文项目）
- **架构**：AI Native 长篇小说生产系统
- **核心创新**：
  - **写法引擎**：从参考文本提取写法特征，保存为可复用资产
  - **卷级工作台**：卷战略 → 卷骨架 → 节奏板 → 拆章 → 章节细化
  - **AI自动导演**：从一句灵感出发自动生成整本方向候选
  - **Prompt Registry**：统一管理所有AI链路的提示词
  - **流派模式控制**：爽感推进、建设经营、关系情感等模式
- **技术栈**：TypeScript + LangChain/LangGraph + Qdrant + SQLite
- **对我们的价值**：**中文语境下的最佳参考**，"写法引擎"概念与我们的风格指南思路吻合

#### ④ novel-multi-agents（⭐10）
- **架构**：5个Agent（大纲专家、角色专家、写作专家、编辑专家、协调者）
- **技术栈**：Python + OpenAI Agent SDK + Ollama（支持本地模型）
- **对我们的价值**：架构简单，适合作为最小可行版本的参考

#### ⑤ terminal-velocity（⭐1.1k）
- **架构**：10个Agent协作写出了10万字完整小说
- **角色**：SpecificationsAgent、ProductionAgent、ManagementAgent、EvaluationAgent等
- **对我们的价值**：证明了多Agent协作写长篇小说的可行性

### 2.2 框架在写作场景的适用性

| 框架 | 写作场景适用度 | 特点 | 推荐度 |
|------|--------------|------|--------|
| **纯Python + 脚本编排** | ⭐⭐⭐⭐⭐ | 最灵活，无框架耦合，适合OpenClaw环境 | **首选** |
| LangChain/LangGraph | ⭐⭐⭐⭐ | 有成熟的RAG链和状态机，中文生态好 | 备选 |
| OpenAI Agent SDK | ⭐⭐⭐ | 好用但绑定OpenAI，不支持本地模型 | 不推荐 |
| CrewAI | ⭐⭐⭐ | 角色定义直观，但中文支持一般 | 可选 |
| AutoGen | ⭐⭐ | 对话驱动，不太适合结构化写作 | 不推荐 |

### 2.3 RAG + LLM 在创意写作中的最佳实践

从调研中提炼的关键实践：

1. **不要全量注入**——按相关性检索上下文，InkOS的JSON delta方案是业界最佳实践
2. **真相文件 > 自由文本**——结构化状态（JSON + schema校验）比markdown更可靠
3. **风格指纹 > 风格描述**——从原文提取统计特征（句长、词频、节奏），注入生成约束
4. **多轮审计 > 一次性生成**——写→审→改循环是保证质量的核心手段
5. **伏笔需要治理系统**——不能只加不收，需要排班、审计、债务追踪

## 三、Agent架构设计

### 3.1 推荐架构：7-Agent管线 + 状态管理系统

基于调研，推荐以下架构（参考InkOS的10-Agent管线，简化为适合我们场景的7-Agent）：

```
用户输入（大纲/已有内容/续写指令）
          │
          ▼
    ┌─────────────┐
    │ 1. 规划Agent │ ← 读取风格指南 + RAG检索 + 真相文件
    │  (Planner)   │    产出：本章意图、必须推进/必须避免
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 2. 编排Agent │ ← 从真相文件中选择相关上下文
    │  (Composer)  │    产出：精简的写作上下文包
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 3. 写作Agent │ ← 风格指纹 + 上下文包 + RAG参考片段
    │  (Writer)    │    产出：章节初稿（3000-5000字）
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 4. 观察Agent │ ← 从正文中提取事实
    │  (Observer)  │    产出：角色、位置、资源、关系、伏笔等增量
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 5. 审计Agent │ ← 对照真相文件 + 风格指南检查
    │  (Auditor)   │    产出：问题列表（严重/一般/建议）
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 6. 修订Agent │ ← 修复审计发现的问题
    │  (Reviser)   │    产出：修订后章节
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │ 7. 状态Agent │ ← 更新真相文件（JSON delta）
    │  (Settler)   │    产出：更新后的世界状态
    └──────┬──────┘
           │
           ▼
     章节完成 + 状态快照
```

### 3.2 核心模块设计

#### 模块A：记忆系统（双层）

**短期记忆（工作记忆）**：
- 当前章节上下文：最后1-2章正文 + 当前章节大纲
- 当前写作意图：本章要推进什么、避免什么
- 上下文窗口限制：控制在2000-4000 token以内

**长期记忆（持久记忆）**：
- 真相文件（JSON）：世界状态、角色矩阵、伏笔系统、章节摘要
- RAG系统：13923个语义向量块，按场景类型分类
- 风格指纹：从原文提取的统计特征

**关键设计决策**：
- 使用JSON而非markdown存储状态（参考InkOS的教训）
- 使用JSON delta而非全量写入（避免上下文膨胀）
- RAG按场景类型（10类）过滤检索结果

#### 模块B：风格控制（三层防线）

| 层级 | 方法 | 作用 |
|------|------|------|
| **第1层：Prompt约束** | 风格指南注入 + 固定表达表 + 禁用词表 | 源头控制 |
| **第2层：RAG参考** | 检索相似场景的原文片段作为写作参考 | 风格锚定 |
| **第3层：审计检测** | AI味检测 + 风格偏离检测 | 事后修正 |

**不推荐fine-tuning**的原因：
- 1100万字数据量虽大，但fine-tuning成本高
- Prompt + RAG方案已有足够好的效果（InkOS验证）
- mimo-v2-pro的1049k上下文足以容纳大量风格参考

#### 模块C：情节管理系统

**真相文件设计（参考InkOS + 自定义）**：

```json
// world_state.json - 世界状态
{
  "currentLocation": "学校天台",
  "activeCharacters": ["主角", "女主角A"],
  "knownFacts": [...],
  "relationships": {...}
}

// hooks.json - 伏笔系统
[
  {
    "id": "hook_001",
    "description": "主角的异体觉醒暗示",
    "plantedChapter": 3,
    "status": "open",  // open/progressing/deferred/resolved
    "resolvedChapter": null,
    "lastAdvancedChapter": 5
  }
]

// chapter_summaries.json - 章节摘要
[
  {
    "chapter": 5,
    "summary": "主角在天台遇到女主角A，触发异体觉醒预兆...",
    "characters": ["主角", "女主角A"],
    "hookChanges": ["hook_001 advanced"]
  }
]

// character_matrix.json - 角色矩阵
{
  "characters": {
    "主角": {
      "personality": "温和、自嘲、关键时果断",
      "abilities": ["异体·丝/幻"],
      "relationships": {"女主角A": "暗恋"},
      "emotionalState": "困惑中带点期待"
    }
  }
}
```

#### 模块D：质量控制

**审计维度（参考InkOS的33维度，精简为核心15维度）**：

1. 角色一致性（性格、口头禅、行为模式）
2. 视角一致性（第一人称/第三人称不混用）
3. 时间线连续性
4. 地点连续性
5. 物资/道具连续性
6. 伏笔回收检查
7. 风格偏离检测（句式、标点、口语化程度）
8. AI味检测（高频词、过度总结、句式单调）
9. 幽默风格检查（是否有自嘲、反差、吐槽）
10. ACG文化引用合理性
11. 对话风格检查
12. 字数达标检查
13. 情感弧线检查
14. 重复内容检测
15. 大纲偏离检查

#### 模块E：长文本生成策略

**分段生成 + 拼接**策略：

```
1. 规划Agent生成本章结构：
   - 开头（500-800字）：场景引入
   - 发展（1000-1500字）：事件展开
   - 高潮（800-1200字）：冲突/转折
   - 结尾（500-800字）：收束/悬念

2. 写作Agent按段生成，每段注入：
   - 本段在结构中的位置
   - 上一段的最后几句话（衔接）
   - 风格参考片段（RAG检索）
   - 必须包含的元素（从大纲）

3. 生成后整体审查连贯性
```

## 七、实施路线图

### 阶段1：基础框架（预估1-2周）

**目标**：跑通单章写作的最小闭环

**产出**：
- LLM调用封装（支持mimo-v2-pro，含重试和错误处理）
- RAG查询接口封装（从现有query_v3.py封装）
- Pydantic状态模型定义（world_state, hooks, characters等）
- 真相文件读写模块（JSON + 增量更新）
- 简单的Writer Agent（能根据大纲 + RAG参考 + 风格指南写一章）

**验证标准**：
- 能根据简单大纲生成一章2000-3000字的内容
- 输出风格与风上忍有基本相似度
- 真相文件能正确记录角色和事件

**依赖**：无（所有资源已就绪）

### 阶段2：质量控制系统（预估1-2周）

**目标**：实现写→审→改的闭环

**产出**：
- Observer Agent（从正文中提取事实更新真相文件）
- Auditor Agent（15维度审计）
- Reviser Agent（根据审计结果修改）
- 写→审→改循环管线

**验证标准**：
- 审计Agent能检测出风格偏离、角色不一致等问题
- 修订后质量有明显提升
- 能自动处理5轮以内的审计修订循环

**依赖**：阶段1完成

### 阶段3：规划与编排系统（预估1-2周）

**目标**：支持多章节连贯写作

**产出**：

## 八、风险和挑战

### 8.1 技术风险

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| **mimo-v2-pro输出质量不稳定** | 章节质量参差不齐 | 多轮审计+修订；备选deepseek-v3 |
| **上下文窗口虽大但成本高** | 长时间运行成本不可控 | 严格控制注入上下文大小；按相关性筛选 |
| **JSON输出格式不稳定** | 状态更新失败 | Pydantic校验 + LLM重试 + 容错解析 |
| **RAG检索不精准** | 参考片段不相关 | 优化检索query；结合场景类型过滤 |
| **风格模仿效果不够好** | 输出不像风上忍 | 三层防线（prompt+RAG+审计）；持续迭代prompt |

### 8.2 产品风险

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| **长篇连贯性不足** | 写到20章后角色跑偏 | 真相文件系统 + 跨章审计 |
| **伏笔只加不收** | 故事结构松散 | hookAgenda排班系统（参考InkOS） |
| **幽默感不够** | 风格失真 | 重点优化自嘲/反差/吐槽的prompt |
| **"AI味"过重** | 读者不适 | AI味检测维度 + 禁用词表 + 修订 |

### 8.3 已知的行业教训（来自InkOS经验）

1. **不要用markdown存状态**——20章后上下文膨胀导致400错误，必须用JSON
2. **不要全量写入**——用JSON delta，否则状态文件越来越大
3. **字数控制不要硬截断**——用Normalizer软纠偏，否则毁章
4. **伏笔需要强制管理**——必须有排班+审计+债务追踪，否则只加不收
5. **多Agent管线必须有重试机制**——任何一步失败都可能导致整章报废

## 九、结论和建议

### 9.1 核心结论

1. **技术可行性高**：我们已有80%的基础资源（小说全文、RAG系统、风格指南、分析报告），核心缺失的是Agent编排框架和状态管理系统。

2. **InkOS是最佳参考架构**：其10-Agent管线、真相文件系统、伏笔治理、审计机制已经经过实践验证。建议"站在巨人肩膀上"，参考其架构设计，用Python重新实现。

3. **不推荐fine-tuning**：mimo-v2-pro的1049k上下文 + RAG + Prompt Engineering 方案已经足够。Fine-tuning成本高、灵活性差。

4. **最小可用版本3-6周可完成**：阶段1-3即可产出能用的写作Agent，阶段4-5是质量提升。
